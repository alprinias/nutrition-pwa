# Nutrition PWA — Project Context

## What this is
A progressive web app (PWA) for daily meal tracking, macro monitoring, and body measurements.
Built from an existing Excel workbook (Διατροφή.xlsx) that the user maintained manually.
Live at: **https://nutrition-pwa-wheat.vercel.app**

## Database schema

### Reference tables (shared, read-only for regular users)
```
ingredients       id, name, category, unit, ref_quantity, calories, protein_g, carbs_g, fat_g, sodium_mg, potassium_mg
recipes           id, name, unit, total_weight_g, servings, calories_total, protein_total, carbs_total, fat_total, calories_per_unit, protein_per_unit, carbs_per_unit, fat_per_unit
recipe_items      id, recipe_id → recipes, ingredient_id → ingredients, quantity, unit
app_config        key (PK), value  ← stores admin_user_id
```

### Per-user tables (Row Level Security on user_id)
```
meal_log          id, user_id, logged_at, recipe_id, ingredient_id, quantity, unit, note, calories, protein_g, carbs_g, fat_g
user_targets      id, user_id, target_kcal, carb_min_g, prot_min_pct, prot_max_pct, carb_min_pct, carb_max_pct, fat_min_pct, fat_max_pct
body_measurements id, user_id, measured_at (date, unique per user), weight_kg, fat_pct, water_pct, note
```

### Key notes
- `meal_log`: either `recipe_id` OR `ingredient_id` is set, not both. Macros are snapshotted at log time.
- `recipes.unit`: `'gr'` = log by grams (calories_per_unit = per 1g), `'pc'` = log by pieces (calories_per_unit = per 1 piece)
- `recipes.servings`: for `pc` recipes this equals the total piece count (e.g. 16 for a pie with 16 pieces)
- `user_targets`: auto-created with defaults when a new user signs up (trigger: `on_auth_user_created`)
- `app_config`: `admin_user_id` key holds the UUID of the admin user (set manually via SQL editor)
- Macro calorie constants: Protein = 4 kcal/g, Carbs = 4 kcal/g, Fat = 9 kcal/g
- All times stored as UTC in Supabase; displayed in local time (el-GR, 24hr format) via toLocaleTimeString with hour12: false
- Times entered via the app use setHours (local time) before toISOString() — correct for Greek timezone (UTC+2/+3)

## Admin system
- Admin is identified by matching `auth.users.id` to `app_config` row where `key = 'admin_user_id'`
- Stored in `useAdminStore` (Pinia) — loaded once on Foods screen mount
- `isAdmin` computed: `loaded && currentUserId === adminUserId`
- Admin sees in Foods: ＋ button (top right), ✏️ Edit + 🗑️ Delete in detail sheet
- Non-admin users see no edit controls — identical read-only experience
- To change admin: update `app_config` row via Supabase SQL editor

## Key behaviours & decisions

### Add Meal screen
- Ingredients: macros shown per `ref_quantity` (usually 100g). Quantity defaults to `ref_quantity`.
- Recipes (gr): quantity defaults to `total_weight_g`, macros scale as `calories_per_unit × qty`
- Recipes (pc): quantity defaults to `servings` (total pieces), macros scale as `calories_per_unit × qty`
- Logged macros are snapshotted at log time so later edits to ingredients don't affect history
- Time input uses `setHours` (local) not `setUTCHours` — critical for correct Greek timezone storage

### Foods CRUD (admin only)
- Ingredient form: name, category (datalist autocomplete), unit, ref_quantity, calories, protein, carbs, fat, sodium
- Recipe form: name, unit (gr/pc), total weight or piece count, ingredient lines with live search + quantity
- Live macro preview updates as ingredient lines are added/changed
- On save, recipe recomputes all `*_total` and `*_per_unit` columns from ingredient lines
- Delete blocked by Postgres foreign key if item is referenced in meal_log or recipe_items — shows friendly message

### Charts
- Y axis: min = max(0, dataMin − 10% of spread), max = dataMax + 10% of spread
- Target line shown as dashed line labelled "T"
- Body charts only appear if body data exists in the selected date range
- Week aggregation uses ISO week numbers; Month uses YYYY-MM
- Dots shown on data points only when ≤ 30 points

### Body measurements
- One entry per user per date (unique constraint: `user_id, measured_at`)
- Upsert on save — same date = update existing
- Tap any history row to jump to that date and edit it
- Paginated at 20 entries with Load more

### Time display
- All times shown in 24hr format using `toLocaleTimeString('el-GR', { hour: '2-digit', minute: '2-digit', hour12: false })`
- Applied in: TodayView, MealLogView, HistoryView

### Password reset flow
1. `/forgot-password` → user enters email → `supabase.auth.resetPasswordForEmail()` with redirectTo `/reset-password`
2. Email link → `/reset-password` → token in URL hash → `supabase.auth.updateUser({ password })`
3. On success: signs out, redirects to login

## Change deployment workflow
```
1. Edit files locally
2. npm run dev  →  test at localhost:5173
3. If DB change needed: run SQL in Supabase first
4. git add . && git commit -m "description" && git push
5. Vercel auto-deploys in ~30-60 seconds
6. Users get update on next app open (service worker autoUpdate)
```

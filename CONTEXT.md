# Nutrition PWA — Project Context

## What this is
A progressive web app (PWA) for daily meal tracking, macro monitoring, and body measurements.
Built from an existing Excel workbook (Διατροφή.xlsx) that the user maintained manually.
Live at: **https://nutrition-pwa-wheat.vercel.app**

## Database schema

### Reference tables (shared, visibility controlled by RLS)
```
ingredients       id, name, category, unit, ref_quantity, calories, protein_g, carbs_g, fat_g,
                  sodium_mg, potassium_mg, owner_user_id, is_public
recipes           id, name, unit, total_weight_g, servings,
                  calories_total, protein_total, carbs_total, fat_total,
                  calories_per_unit, protein_per_unit, carbs_per_unit, fat_per_unit,
                  owner_user_id, is_public
recipe_items      id, recipe_id → recipes, ingredient_id → ingredients, quantity, unit
app_config        key (PK), value  ← stores admin_user_id
moderation_requests  id, user_id, entity_type, entity_id, status, note, admin_note,
                     created_at, reviewed_at, reviewed_by
```

### Per-user tables (Row Level Security on user_id)
```
meal_log          id, user_id, logged_at, recipe_id, ingredient_id, quantity, unit, note,
                  calories, protein_g, carbs_g, fat_g
user_targets      id, user_id, target_kcal, prot_min_g, carb_min_g,
                  prot_min_pct, prot_max_pct, carb_min_pct, carb_max_pct,
                  fat_min_pct, fat_max_pct, show_public_foods
body_measurements id, user_id, measured_at (date, unique per user), weight_kg, fat_pct, water_pct, note
```

### Key notes
- `meal_log`: either `recipe_id` OR `ingredient_id` is set, not both. Macros snapshotted at log time.
- `recipes.unit`: `'gr'` = log by grams, `'pc'` = log by pieces
- `recipes.servings`: for `pc` recipes equals total piece count
- `user_targets` defaults: target_kcal=1850, prot_min_g=120, carb_min_g=135
- `app_config`: `admin_user_id` key holds admin UUID — set manually via SQL editor
- `ingredients.owner_user_id`: all migrated rows set to admin UUID; new user items set to their UUID
- `ingredients.is_public`: migrated rows = true; new user items = false (private by default)
- `moderation_requests.status`: 'pending' | 'approved' | 'rejected'
- Macro calorie constants: Protein = 4 kcal/g, Carbs = 4 kcal/g, Fat = 9 kcal/g
- All times stored as UTC; displayed in local time (el-GR, 24hr, hour12: false)
- Times entered via app use setHours (local) before toISOString()
- All dates displayed as dd/mm/yyyy

## TodayView layout (three-zone)
```
┌─────────────────────────────┐  ← AppShell sticky header
│  Calorie ring + macro bars  │  ← shrink-0, never scrolls
├─────────────────────────────┤
│   Meal list (scrollable)    │  ← flex-1 overflow-y-auto min-h-0
├─────────────────────────────┤
│     + Add meal button       │  ← shrink-0, pb-20 clears fixed nav
└─────────────────────────────┘  ← AppShell fixed bottom nav (4 tabs, grid-cols-4)
```
- AppShell `<main>` uses `overflow-hidden flex flex-col` on Today route only
- `min-h-0` on middle zone critical for correct scroll behaviour
- Meal list shows: time, name, qty+unit, P/C/F, kcal, delete (✕) with confirmation sheet
- Macro totals shown inline in meals header

## HistoryView day detail
- Calorie ring (w-20) on the left, macro bars on the right — same pattern as TodayView
- Ring fill = dayTotals.calories / targets.target_kcal
- Each meal row has ✏️ edit and ✕ delete buttons
- Edit sheet: quantity + time + note inputs; live macro preview scales proportionally
- Delete sheet: confirmation before removing; updates monthData cache so dot disappears if day empty
- "Add meal for [date]" button navigates to `/add?date=YYYY-MM-DD`
- AddMealView reads `route.query.date`, shows amber banner when logging for a past date
- After logging from History, redirects back to History (not Today)

## Targets system
- `target_kcal`: daily calorie target (default 1850)
- `prot_min_g`: minimum daily protein grams (default 120) — protein bar max in Today/History/Charts
- `carb_min_g`: minimum daily carbs grams (default 135)
- Carbs/Fat bars use: percentage × target_kcal / caloric density

## Admin system
- Admin identified by `app_config` where `key = 'admin_user_id'`
- `useAdminStore` (Pinia) — loaded once per session
- Admin sees edit/delete on all Foods items; Admin link in Settings

---

## Visibility system (ingredients & recipes)
- New items: `is_public = false`, `owner_user_id = current user`
- Promoted to public only by admin approval
- Once public: owner cannot edit (offered "Create private copy" instead)
- `show_public_foods` toggle in Settings filters Foods browser and Add Meal search

### Moderation flow
1. Owner submits private item for review → `moderation_requests` row (status='pending')
2. Admin approves → `is_public = true` (item locked) or rejects with note

## Key behaviours

### Add Meal
- Reads `?date=` query param — used when navigating from History
- Shows amber banner when logging for a past date
- After save: redirects to History if `?date=` present, else Today
- Macros snapshotted at log time

### Edit meal (History)
- Quantity change scales all macros proportionally
- Time can be changed within the same day
- Note can be added/edited

### Foods CRUD
- New items always private; recipe save recomputes all macro columns
- Delete blocked by Postgres FK if referenced in meal_log or recipe_items

### Charts
- Y axis: min = max(0, dataMin − 10% of spread), max = dataMax + 10% of spread
- Body charts appear only if body data exists in range
- Week: ISO week; Month: YYYY-MM

### PWA updates
- `skipWaiting + clientsClaim` → instant activation on deploy

### Date & time display
- Times: 24hr `hour12: false`
- Dates: `{ day: '2-digit', month: '2-digit', year: 'numeric' }` → dd/mm/yyyy
- Month headers: `{ month: 'long', year: 'numeric' }` → "Μάιος 2026"
